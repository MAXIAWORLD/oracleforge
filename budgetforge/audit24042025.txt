  Ce qui est solide

  1. SSRF proxy : les URLs providers sont hardcodées dans proxy_forwarder.py. X-Provider-Key n'est jamais interprétée
  comme URL. OK.
  2. Signature Stripe webhook : stripe.Webhook.construct_event est utilisé correctement, 400 sur mismatch.
  3. SQL injection : 100% via ORM SQLAlchemy avec filter(Column == value). Aucune raw query détectée.
  4. Isolation via bf-key : un client ne peut PAS lire les projets d'un autre via sa bf-xxx — elle sert au proxy, pas
  aux APIs admin/portal.
  5. Portal magic link : token 256 bits, TTL 1h, usage unique (delete on verify), HTTPOnly+SameSite=lax+Secure si https.

  Correctifs prioritaires

  1. distributed_budget_lock.py + budget_lock.py (CRITIQUE)
  → Avec 2 workers uvicorn sans Redis, la sérialisation inter-process n'existe pas.
  → Option A : installer Redis (obligatoire en prod multi-worker).
  → Option B : remplacer le lock asyncio par un verrou SQL : SELECT ... FOR UPDATE sur la row project AVANT le check, ou
   advisory lock SQLite via BEGIN IMMEDIATE. SQLite WAL + busy_timeout=30000 + transaction IMMEDIATE sérialisent les
  écritures entre processus.
  → Option C : passer à 1 worker uvicorn (dégrade le throughput mais élimine la classe entière).
  → Ajouter CHECK SQL côté DB : trigger qui refuse un INSERT si la somme dépasserait budget, atomique avec le SELECT de
  sommation — zéro race possible.

  2. core/config.py + portal.py + core/auth.py (CRITIQUE)
  → Ajouter au startup de app.py : si app_env=="production" ET (admin_api_key=="" OR portal_secret=="" OR
  stripe_webhook_secret=="") → raise à l'import, refus de boot. Aucun dev mode silencieux en prod possible.
  → Supprimer le fallback "portal-dev-secret". En dev, exiger que l'env var soit posée explicitement même à "dev".

  3. proxy_dispatcher.py:239-240 + cost_calculator.py:76 (CRITIQUE)
  → Si UnknownModelError lors de prebill → refuser l'appel avec HTTP 400 "Unknown model, configure pricing first", pas
  cost=0.0.
  → Remplacer except (ValueError, Exception) par except UnknownModelError as e: raise (ou logging warning explicite).
  → Les provider APIs ne valident pas tous le nom de modèle comme invalide (certains proxy-routent) — la compta doit
  refuser par défaut.

  4. services/cost_calculator.py:76 et dynamic_pricing.py (CRITIQUE)
  → Remplacer except (ValueError, Exception) par exceptions spécifiques. Laisser les vraies pannes (réseau, timeout
  httpx) remonter pour qu'on ne serve pas à $0.

  5. Rate limiting asymétrique (HAUT)
  → Appliquer @limiter.limit("30/minute;1000/hour") sur TOUS les endpoints proxy, pas juste /proxy/openai.
  → Ajouter une key_func qui combine IP ET bf-api-key : lambda req: req.headers.get("authorization","") or
  get_remote_address(req). Empêche rotation IP de contourner.

  6. Signup anti-abuse (HAUT)
  → Ajouter un captcha (hCaptcha/Turnstile) sur /api/signup/free. 3/j/IP est ridicule face à un VPN.
  → Ou limiter par hash(email+domaine) : pas plus de 3 comptes par domaine sans validation humaine.

  7. Stripe idempotence + réconciliation (HAUT)
  → Table processed_stripe_events(event_id PRIMARY KEY, created_at). Au début de /webhook/stripe, INSERT OR IGNORE puis
  vérifier rowcount — skip si déjà vu.
  → Sur /success?session_id=X, fetch stripe.checkout.Session.retrieve(X) côté backend et créer le projet si absent
  (idempotent). Ne plus dépendre uniquement du webhook.

  8. Portal cookie révocable (HAUT)
  → Ajouter iat (issued-at) dans le payload signé : email.iat.sig. Refuser si iat < portal_session_invalidated_at
  (colonne SiteSetting globale, ou per-user).
  → Ou stocker chaque session dans DB avec un id révocable, et mettre l'id dans le cookie (signé).

  9. check_provider fail-open sur JSON corrompu (HAUT)
  → except json.JSONDecodeError: raise HTTPException(500, "Project config corrupted") plutôt que return. Même chose pour
   downgrade_chain. Fail-closed, pas fail-open sur décision de sécurité.

  10. SQLite locked 500 (HAUT)
  → create_engine(..., connect_args={"check_same_thread": False, "timeout": 30}) + PRAGMA journal_mode=WAL au startup.
  Middleware FastAPI qui attrape OperationalError et retry 3× avec jitter avant de 503.

  11. SSRF DNS rebinding (MOYEN→HAUT si client sensible)
  → Re-résoudre l'hostname au moment de l'envoi du webhook, vérifier que l'IP résolue n'est pas dans _BLOCKED_NETWORKS
  juste avant le httpx.post. Coût : 1 DNS lookup par alerte.

  12. Stripe webhook perdu (HAUT)
  → Sur /success?session_id=X, faire un appel côté backend qui appelle stripe.checkout.Session.retrieve(X), vérifie
  payment_status=="paid", et upsert le projet si absent. Le client paye, la page /success GARANTIT la création, le
  webhook devient juste un fallback async.

  Ship-blockers absolus : 1, 2, 3, 4. Tant que ces 4 ne sont pas fixés, le produit n'est pas prod-ready.
