**Status**: shipped 2026-04-17.
**Scope**: a 6th independent price source — Uniswap v3 time-weighted
average prices (TWAPs) read directly from curated high-liquidity pools
on Base and Ethereum mainnet via `observe(uint32[])` eth_call. Dedicated
REST route, 11th MCP tool, 12th method on both SDKs, 11th tool on every
Python plugin, 11th action on the Eliza plugin.
**Effort**: ~1 day.
**Breaking changes**: **none**.

## 1. Design rationale

### Why TWAP?

A TWAP derived from a Uniswap v3 pool is a price signal that is
independently verifiable on-chain AND robust to short-term manipulation.
Moving the TWAP by 1% requires keeping the pool at a skewed price for
the entire window — on a high-liquidity pool (WETH/USDC 0.05% at $150M+
TVL) that costs millions of dollars in capital parked on the wrong side
of the order book for 30 minutes.

That makes TWAP structurally different from the other on-chain source
we already have (Chainlink aggregators, which are heartbeat-plus-
deviation push oracles maintained by a private set of validators). The
two signals complement each other: a divergence between Chainlink spot
and Uniswap v3 TWAP is a decent "something is weird" signal.

### Why only 3 pools?

The scratch audit (2026-04-16) probed 8 candidate pools. Three passed:

| Symbol | Chain    | Pool                                           | Fee   | TWAP agreement |
|--------|----------|------------------------------------------------|-------|----------------|
| ETH    | ethereum | 0x88e6A0c2dDD26FEEb64F039a2c41296FcB3f5640     | 0.05% | <0.05% vs Pyth |
| ETH    | base     | 0xd0b53D9277642d899DF5C87A3966A349A798F224     | 0.05% | <0.05% vs Pyth |
| BTC    | ethereum | 0x99ac8cA7087fA4A2A1FB6357269965A2014ABc35     | 0.30% | <0.1% vs Pyth  |

Five were dropped:

- **cbBTC/USDC on Base**: the candidate address I tried returned garbage,
  meaning I had the wrong pool — rather than guessing, dropped from V1.5.
  Can be re-added once the canonical pool is verified.
- **LINK / UNI / AAVE on Ethereum** (all quoted in WETH): pools exist but
  TVL is thin enough that the TWAP diverges 20-30% from the spot price
  reported by Pyth/Chainlink. Uniswap is not the canonical venue for
  those assets in 2026.
- **WBTC/WETH** used in the audit for sanity checks only — the price is
  derived (WBTC/WETH × ETH/USD), not a direct USD quote, and adding a
  derived feed to the public surface would invite confusion.

V1.5 prioritizes **3 solid pools over 8 mediocre ones**. Extending the
catalog is a non-breaking addition any time a high-liquidity canonical
pool becomes available — see `scripts/precompute_pyth_solana_pdas.py`
for the pattern to replicate.

## 2. Decisions (D1–D12)

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | No new runtime dependency. Pool addresses hardcoded; `observe()` calldata is ABI-encoded manually (no `eth-abi` / `web3.py` needed). | Keeps the install footprint at zero change; observe() ABI is stable. |
| D2 | Shipping chains: `base`, `ethereum` only. Arbitrum, Optimism etc. can come in V1.6 if there is demand and canonical TVL on those chains. | Matches the V1.1 Chainlink surface for stability — two chains both already have configured RPC pools. |
| D3 | Default window = 1800 s (30 min). Range [60, 86400]. | 30 min is the Uniswap-recommended default for oracle reads (see Uniswap Oracle Whitepaper); the range covers both short (arbitrage detection) and long (daily average) use cases. |
| D4 | Cache TTL = 60 s. A 30-min TWAP moves a few basis points per minute, so caching for 1 min is almost free. | Avoids hammering public RPCs on tight polling loops. |
| D5 | Per-chain circuit breaker (5 failures / 60 s cooldown). Ethereum and Base fail independently. | A bad Ethereum RPC must not disable Base. |
| D6 | Pool `observe()` revert does NOT trip the breaker. The pool answered; it rejected the window as too wide for its observation cardinality. The breaker is for RPC health, not pool config. | Defence against breaker-poisoning by querying a legit-but-underused pool. |
| D7 | Reject verification_level other than Full? **N/A** — Uniswap v3 pools don't have Pyth's Wormhole-style verification. The TWAP math itself is deterministic. |
| D8 | Route: `GET /api/twap/{symbol}?chain=base\|ethereum&window=1800`. X-API-Key + daily quota. | Mirror of `/api/chainlink/{symbol}` with the extra `window` parameter. |
| D9 | MCP tool: `get_twap_onchain` (11th tool). 3 parameters: `symbol`, `chain` (default `ethereum`), `window_s` (default 1800). | Same name on every transport (REST query, MCP, SDK, plugins). |
| D10 | Python SDK `client.twap(symbol, chain, window_s)` + TypeScript SDK `client.twap(symbol, chain, windowSeconds)`. Bump `0.3.0 → 0.4.0`. | Parity Python ↔ TypeScript. |
| D11 | 4 Python plugins add `MaxiaOracleGetTwapTool` (11th tool each). Eliza plugin adds `GET_TWAP_ONCHAIN` action with a smart `extractWindow()` that parses "30 minutes" / "1 hour" / "3600 seconds" from the prompt. Bumps: Python plugins `0.3.0 → 0.4.0`, Eliza `0.2.0 → 0.3.0`. | Full parity across every framework. |
| D12 | MCP `SERVER_VERSION = 0.1.5`. Git tag `v0.1.5`. | Semver synchronous with the rest of the packages. |

## 3. Math reference

Uniswap v3 pools expose `observe(uint32[] secondsAgos)` which returns
two arrays: `(int56[] tickCumulatives, uint160[] secondsPerLiquidityCumulativeX128)`.
We call `observe([window, 0])` — one old timestamp, one current — and
take the slope:

```
avg_tick = truncate_toward_zero((tickCumulatives[1] - tickCumulatives[0]) / window)
```

The price of token1 in terms of token0 at that tick is `1.0001 ** tick`.
Converting to a human-readable number requires accounting for each
token's decimals:

```
human_price(1 token0 in token1 units) = 1.0001^tick * 10^(decimals0 - decimals1)
```

When the asset we want to PRICE is `token1` (e.g. on the Ethereum
USDC/WETH pool where USDC < WETH lexicographically, so USDC is token0),
we invert:

```
human_price = 1 / (1.0001^tick * 10^(decimals0 - decimals1))
```

That inversion is driven by the `base_is_token0` flag on each pool
entry in `UNISWAP_V3_POOLS`.

**Solidity integer division caveat**: Solidity's `int / int` truncates
toward zero, while Python's `//` floors toward negative infinity. On a
negative tick this produces a 1-LSB difference (e.g. -198736 vs -198737
in the V1.5 audit). We explicitly truncate toward zero in
`_twap_price_from_ticks` to match what any on-chain consumer would see.

## 7. Non-goals

- **No derived feeds.** A pool like WBTC/WETH is read-only for sanity
  checks; it is not exposed on the public surface because the USD price
  would be computed as `WBTC/WETH × ETH/USD` and that composition
  invites confusion about the divergence metric.
- **No multi-source aggregation** with the TWAP. The TWAP is intentionally
  single-source: the whole point is that an agent can compare
  `/api/price/ETH` (median) with `/api/twap/ETH` (on-chain) as two
  independent anchors. Mixing them into the median would erase that.
- **No liquidity-weighted TWAP.** The Uniswap v3 observe() also returns
  `secondsPerLiquidityCumulativeX128`, which enables a liquidity-
  weighted variant. Adding it to the payload is a non-breaking change
  any time a user asks, but it's not shipped in V1.5.
- **No write path.** MAXIA Oracle is read-only; V1.5 is pure eth_call.

## 8. Follow-ups (future V1.x)

- **Add more pools** once the canonical high-TVL pool per asset is
  verified on-chain. Top of the list: cbBTC/USDC on Base,
  WETH/USDT on Ethereum, WETH/WBTC on Base when it accumulates TVL.
  Procedure: query the pool, replay the audit script, confirm the TWAP
  agrees with other oracles to < 0.5 %, add the entry to
  `UNISWAP_V3_POOLS` with the `tvl_note` field.
- **Add Arbitrum** when Uniswap v3 pools there reach comparable TVL.
  The V1.1 Chainlink reader already uses Arbitrum RPCs, so the RPC
  pool is already configured — only the pool catalog would grow.
- **Liquidity-weighted TWAP** variant exposed as `variant=liq_weighted`
  query param. Opt-in, does not change the default response shape.
- **EMA-style TWAP** (exponential moving average) if a user has a
  concrete use case — Uniswap v3 doesn't expose it natively, would
  require post-processing multiple observe() points client-side.
