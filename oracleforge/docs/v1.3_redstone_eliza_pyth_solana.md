# MAXIA Oracle — V1.3 : RedStone + plugin Eliza (Pyth Solana reporté V1.4)

**Status** : shipped 2026-04-16.
**Scope initial** : 3 ajouts non-breaking (RedStone, Pyth Solana, plugin Eliza).
**Scope final shippé** :
1. **RedStone** en 4e source d'upstream dans `services/oracle/`. ✅
2. **Plugin Eliza** (`eliza-plugin-maxia-oracle`, TS) — 5e plugin framework. ✅
3. ~~Pyth natif Solana (lecture on-chain)~~ → **reporté V1.4** après audit live.

**Pourquoi Pyth Solana est reporté** : l'impl V1.3 a été écrite complète (module binaire, route, MCP tool, SDKs, plugins) puis testée **live** contre Solana mainnet-beta avant commit. Résultat : les Pyth V2 price accounts listés dans la doc officielle (ex: `GVXRSBj...UoMU` pour BTC) renvoient `status=0` (unknown) et `slot` figé depuis ~23 jours. Les publishers ont migré vers le programme `Pyth Solana Receiver` avec un nouveau layout (`PriceUpdateV2`, accounts PDA-dérivés du feed_id Hermes) — incompatible avec le décodeur V2 écrit. Shipper ça = shipper stale. Decodeur + dict feeds + tests retirés. À refaire en V1.4 avec le Receiver program (feed_id → PDA derivation, layout différent).

**Effort réel** : ~1 jour (audit + RedStone + Eliza + cleanup Pyth Solana).
**Breaking changes** : **aucun**. Rétrocompat stricte — un client V1.0/V1.1/V1.2 continue de fonctionner à l'identique.

## 1. Décisions figées

### 1.1 RedStone (4e source)

| # | Sujet | Décision | Motivation |
|---|---|---|---|
| R1 | Endpoint | `https://api.redstone.finance/prices?symbol={SYM}&provider=redstone-primary-prod&limit=1`. Public gratuit, pas d'API key. | RedStone primary-prod est le provider officiel le plus stable, coverage large (400+ symbols). |
| R2 | Coverage | Ajout automatique dans `multi_source.collect_sources()` pour **tout symbol** (RedStone couvre crypto majors + long-tail + forex + equities). Pas de feed dict hardcodé — on tente l'appel, on drop silencieux si `symbol not found`. | RedStone renvoie 404 si inconnu. Pas besoin de maintenir un dict dupliqué. Coverage réelle > coverage déclarée. |
| R3 | Cache TTL | 10s (même ordre que Pyth Hermes cache 5s normal + marge pour l'endpoint REST public). | Rate-limiter côté serveur + budget latence agent. 10s = compromis raisonnable. |
| R4 | Shape retour | Même schema que `get_chainlink_price` : `{price, publish_time, age_s, stale, source: "redstone", symbol}`. Stale = `age_s > 300` (RedStone publish every ~10s, 5min = dead feed). | Uniformité cross-source = simplifie `multi_source.collect_sources()` et `divergence`. |
| R5 | Endpoint REST dédié | `GET /api/redstone/{symbol}` (nouveau), parallèle à `/api/chainlink/{symbol}`. Auth X-API-Key + rate-limit standard. | Donne un accès single-source déterministe pour audit/cross-check, comme Chainlink V1.1. |
| R6 | MCP tool | Nouveau tool `get_redstone_price(symbol)` (9e tool MCP). | Parité avec l'ajout REST — un agent MCP a la même surface qu'un agent REST. |
| R7 | SDK (Python + TS) | Nouvelle méthode `redstone(symbol)` (parité 2 SDKs). | Parité stricte SDK/MCP/REST, règle établie depuis V1.1. |
| R8 | Plugins | Nouveau tool `MaxiaOracleGetRedstoneTool` ajouté dans les **4 plugins existants** (langchain, autogen, crewai, llama-index). | Un symbol supporté doit l'être sur toute la surface. Pas de plugin drift. |

### 1.2 Pyth Solana natif (lecture on-chain)

| # | Sujet | Décision | Motivation |
|---|---|---|---|
| S1 | Mécanisme | Lire directement les Pyth **price accounts** sur Solana mainnet via `getAccountInfo` JSON-RPC + décodage binaire du layout v2. Pas d'appel Hermes. | "Pyth natif Solana" = vérifiable on-chain, sub-second, 400+ feeds. Hermes = relais HTTP. Les deux coexistent. |
| S2 | Layout decoder | Parseur binaire inline (pas de dep `pythclient`). Offsets Pyth V2 `PriceAccount` connus et stables depuis 2022. ~80 lignes de struct unpack. | Ajouter une dep pour 80 lignes de binary unpack = overkill. `pythclient` traîne `solana-py` + `construct` + ~10 sub-deps. On reste sur stdlib `struct`. |
| S3 | RPC pool | Réutilise `SOLANA_RPC_URLS` (Chainstack → Alchemy → Helius → SOLANA_RPC custom → 3 publics). Déjà configuré dans `core/config.py`. | Zéro dep nouvelle, résilience existante. |
| S4 | Mapping symbol → account | Nouveau dict `PYTH_SOLANA_ACCOUNTS` dans `pyth_solana_oracle.py` — 30-50 symbols majeurs (crypto + equity) copiés depuis https://pyth.network/developers/price-feed-ids#solana-mainnet. Feeds vérifiés à l'import par `description` account (via product_account prev hop) — non, trop cher. **Simple sanity check** : `account_type == 3` (price) + `ver == 2` + `magic == 0xa1b2c3d4`. | Liste officielle Pyth est stable. Sanity check binaire au runtime rejette un account corrompu ou wrong-type sans freezer le boot. |
| S5 | Shape retour | `{price, confidence, confidence_pct, publish_time, age_s, stale, source: "pyth_solana", symbol, price_account, slot}`. Staleness = `age_s > 60` (Pyth Solana publish every 400ms, 60s = dead ou slot lag). `slot` ajouté pour debug. | Consistent avec `get_pyth_price` + 2 champs solana-specific (`price_account`, `slot`). |
| S6 | Endpoint REST dédié | `GET /api/pyth/solana/{symbol}` (nouveau), auth X-API-Key + rate-limit standard. | Single-source audit, parallèle à Chainlink et RedStone. |
| S7 | multi_source | **Pas d'ajout automatique dans `collect_sources()`**. Pyth Solana reste single-source on-demand. | Pyth Hermes est déjà dans `collect_sources()`. Ajouter Solana-on-chain dupliquerait la même source underlying (même publishers). Divergence ne serait pas indépendante — fausserait la métrique. Règle établie V1.1 D5 : "agrégation multi-chain d'un même feed n'a pas de sens sémantique". |
| S8 | MCP tool | Nouveau tool `get_pyth_solana_onchain(symbol)` (10e tool MCP). | Parité REST/MCP. |
| S9 | SDK (Python + TS) | Nouvelle méthode `pythSolana(symbol)` / `pyth_solana(symbol)`. | Parité. |
| S10 | Plugins | Nouveau tool `MaxiaOracleGetPythSolanaTool` dans les 4 plugins existants. | Parité. |

### 1.3 Plugin Eliza (TS)

| # | Sujet | Décision | Motivation |
|---|---|---|---|
| E1 | Package name | `eliza-plugin-maxia-oracle` (même convention que `langchain-maxia-oracle`). Publié sur npm sous `@maxia/eliza-plugin-oracle` ? Décision : **non-scoped `eliza-plugin-maxia-oracle`** pour découvrabilité via `npm search eliza-plugin-*`. | ElizaOS communauté utilise ce préfixe (eliza-plugin-\*). Scoper casserait la discovery. |
| E2 | Dépendance SDK | `@maxia/oracle` (SDK TS existant). Peer dep `@elizaos/core` (pas runtime dep pour rester léger). | Le SDK TS est déjà publié v0.1.0. Eliza est juste un adapter. |
| E3 | Surface | **10 actions** Eliza : 8 actions mirror des 8 MCP tools existants + 2 nouvelles (V1.3) `GET_REDSTONE_PRICE`, `GET_PYTH_SOLANA_ONCHAIN`. | Parité stricte avec la surface MCP (9 tools + RedStone + Pyth Solana = 10). Un agent Eliza voit la même chose qu'un agent MCP. |
| E4 | Pattern Eliza | Export `maxiaOraclePlugin: Plugin` avec `actions: Action[]`. Chaque action `validate()` + `handler()` + `examples` + `description`. Framework Eliza v1.x. | Pattern officiel ElizaOS documenté. Pas d'invention. |
| E5 | Auth | Action `handler()` lit `runtime.getSetting("MAXIA_ORACLE_API_KEY")` ou `process.env.MAXIA_ORACLE_API_KEY`. Jamais en clair dans le plugin. | Pattern standard Eliza pour secrets. |
| E6 | x402 | V1.3 **n'expose pas x402** dans le plugin Eliza (ni sur aucun SDK/plugin). L'agent paie soit via X-API-Key (free tier), soit envoie lui-même une tx x402 et passe le tx_hash dans `X-Payment` → responsabilité agent. | x402 = capacité protocolaire du **serveur**. L'SDK/plugin reste simple (juste un client HTTP). Ajouter x402 client-side = session-signing logic = scope creep. |
| E7 | Tests | Smoke test Node : instancie le plugin, invoque chaque action avec un mock fetch, vérifie shape retour. Pas de test live dans CI (nécessite backend running). | Pattern déjà utilisé pour le SDK TS (`test/client.test.ts`). |
| E8 | TypeScript strict | `strict: true`, pas de `any` hors signature framework (Eliza types pas tous exportés). ESM only (aligné sur `@maxia/oracle`). | Convention projet (CLAUDE.md). |

Layout Pyth V2 price account (simplifié, offsets en bytes) :
```
0x00  magic   u32  = 0xa1b2c3d4
0x04  ver     u32  = 2
0x08  atype   u32  = 3 (price)
0x10  ptype   u32
0x14  expo    i32
0x28  prev_slot     u64
0x30  prev_price    i64
0x38  prev_conf     u64
0xd0  agg.price     i64
0xd8  agg.conf      u64
0xe0  agg.status    u32  (1=trading, autres=rejet)
0xe4  agg.pub_slot  u64
0x128 timestamp     i64  (unix seconds, si présent)
```
Source : https://github.com/pyth-network/pyth-client/blob/main/program/rust/src/c_oracle_header.rs. Pin commit hash dans le commentaire code.

## 9. Risques identifiés et mitigations

| Risque | Probabilité | Mitigation |
|---|---|---|
| API RedStone down ou rate-limit public | Moyenne | Circuit breaker 5 échecs → skip 60s. Impact isolé (une source sur 4 dans multi-source). |
| Layout Pyth V2 changé | Faible (stable depuis 2022, cassé = casse tout Pyth ecosystem) | Magic + ver check en tête du décodeur, erreur claire `invalid price account v2`. Commit hash pythnetwork/pyth-client pinné en commentaire. |
| Solana public RPCs rate-limitent | Forte sans clé dédiée | Cache 5s + fallback Chainstack/Alchemy/Helius déjà configuré core/config.py. Mêmes garde-fous que price_oracle.py. |
| Plugin Eliza : API framework change | Faible (v1.x stable depuis 2025-Q3) | Peer dep `@elizaos/core` sans version range stricte — user choisit sa version. Pattern Action documenté = standard. |
| Divergence multi-source avec RedStone bruyante | Moyenne (RedStone peut différer de 0.5-2% sur illiquides) | `divergence_pct` devient un **meilleur** signal qu'avant (4 sources > 3). Ce n'est pas un bug, c'est l'utilité. |

## 10. Ce qui n'est PAS dans V1.3

**Hors scope explicite** :
- x402 Solana (attente spec officielle Coinbase, cf. roadmap V1.5).
- Uniswap v3 TWAP on-chain (V1.4).
- Nouveau plugin AI framework autre qu'Eliza (pas de ai16z-specific, pas de Virtuals-specific — Eliza couvre ces écosystèmes Solana).
- Dashboard Solana (MAXIA Oracle V1 = pas de dashboard, règle figée dans CLAUDE.md).
- Ajout de chaînes Chainlink supplémentaires (3 chaînes = assez pour V1).
- Changements à `/api/price/` shape de réponse (strictement additif).
- Refactor du layer cache ou auth (V1.x = extensions non-breaking uniquement).

**Si quelque chose manque ici, c'est qu'on ne le fait pas en V1.3.**
