# MAXIA Oracle — V1.4: Pyth native Solana on-chain reader

**Status**: shipped 2026-04-16.
**Scope**: single non-breaking addition — a 5th independent price source
(Pyth Price Feed Accounts on Solana mainnet, shard 0 sponsored feeds)
exposed through a dedicated REST route, a 10th MCP tool, and a new
method on both SDKs and every plugin.
**Effort**: ~1 day.
**Breaking changes**: **none**. V1.0/V1.1/V1.2/V1.3 clients keep working.

V1.3 attempted this integration using the legacy Pyth V2 `PriceAccount`
layout. Live audit showed those accounts were decommissioned on
mainnet-beta (status=0, slot frozen ~23 days). The implementation was
pulled before ship and rescheduled to V1.4 with the `Pyth Solana
Receiver` / `Push Oracle` architecture. See the post-mortem memory
`reference_pyth_solana_receiver.md` and the V1.3 plan notes.

## 2. Design decisions (D1–D11)

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | **No new runtime dependency.** All PDA addresses are pre-computed offline (`scripts/precompute_pyth_solana_pdas.py`, uses `solders` as a dev tool only) and hardcoded in `PYTH_SOLANA_FEEDS`. | Adding `solders` to the backend would drag in a ~30 MB Rust binary. The approach mirrors `CHAINLINK_FEEDS` V1.1. |
| D2 | **Inline binary decoder** (~60 lines of `struct.unpack_from`). No `pythclient` / `anchorpy` dep. | Stable layout, 134 bytes, well understood. |
| D3 | **Reject any `verification_level != Full`.** Partial verifications only require a subset of Wormhole guardians — lower honesty threshold, unsafe for a disclaimed data feed. | Defence in depth. |
| D4 | Cache TTL = 5 s (12 Solana slots). Stale threshold = 60 s (~150 missed slots). | Matches Pyth's 400 ms target cadence; 60 s is dead-feed territory. |
| D5 | **Not added to `multi_source.collect_sources()`.** Pyth is already reached via Hermes there — same publishers. | Would double-weight the same opinion in the divergence metric. Same rule as V1.3 S7 for Solana. |
| D6 | Dedicated REST route `GET /api/pyth/solana/{symbol}`, X-API-Key + daily quota. | Mirror of `/api/chainlink/{symbol}` and `/api/redstone/{symbol}` for single-source audit calls. |
| D7 | MCP tool `get_pyth_solana_onchain` (10th tool). | Parity REST ↔ MCP. |
| D8 | Python SDK `client.pyth_solana(symbol)` + TypeScript SDK `client.pythSolana(symbol)`. 11th method. | Parity Python ↔ TypeScript. |
| D9 | 4 Python plugins add `MaxiaOracleGetPythSolanaTool` (10th tool each). Eliza plugin adds the `GET_PYTH_SOLANA_ONCHAIN` action (10th action). | Parity across every framework. |
| D10 | Tests: fixture builds a canonical 134-byte blob in memory, 27 offline unit tests in the backend alone, plus SDK/plugin tests. | Zero live network in CI; live smoke run separately before commit. |
| D11 | Version bumps: Python SDK `0.2.0 → 0.3.0`, TS SDK `0.2.0 → 0.3.0`, 4 Python plugins `0.2.0 → 0.3.0`, Eliza plugin `0.1.0 → 0.2.0`, MCP `SERVER_VERSION = 0.1.4`. Git tag `v0.1.4`. | Package catalogue stays aligned with the backend feature level. |

## 3. Protocol reference (verbatim from pyth-crosschain source)

### Program IDs (confirmed via the JS SDK source)

```
Push Oracle (holds Price Feed Accounts):  pythWSnswVUd12oZpeFP8e9CVaEqJg25g1Vtc2biRsT
Solana Receiver (posts ephemeral updates): rec5EKMGg6MxZYaMdyBfgwp4d5rB9T1VQH5pJv5LtFJ
Wormhole Core (guardian set):              HDwcJBJXjL9FpJ7UBsYBtaDjsBUhuLCUYoz3zr8SWWaQ
```

MAXIA Oracle V1.4 reads from the **Push Oracle**. The Receiver program
ID from the V1.3 notes was a red herring — the Receiver creates
ephemeral accounts at fresh keypairs each post, which are useless for
an external reader. The Push Oracle exposes stable PDAs instead.

### PDA derivation

```ts
PublicKey.findProgramAddressSync(
  [shard_u16_little_endian, feed_id_32_bytes],
  pythWSns...,
)
```

MAXIA Oracle always passes `shard_id = 0` (sponsored by the Pyth Data
Association). The feed_id is the same 32-byte identifier used by
Hermes REST.

### Account layout (`PriceUpdateV2`, 134 bytes)

```
[0..8]    anchor discriminator = 0x22F123639D7EF4CD
[8..40]   write_authority                         -- Pubkey (32 B)
[40]      verification_level variant
            0 = Partial (followed by 1 byte num_signatures)
            1 = Full    (no payload)
then (offset depends on variant):
          feed_id                                  -- [u8;32]
          price              i64 little-endian
          conf               u64 little-endian
          exponent           i32 little-endian
          publish_time       i64 little-endian    -- unix seconds
          prev_publish_time  i64 little-endian
          ema_price          i64 little-endian
          ema_conf           u64 little-endian
          posted_slot        u64 little-endian
```

Allocated size is always 134 bytes (max across variants). A Full
update leaves 1 byte of trailing padding we ignore.

Source pinned to
`pyth-network/pyth-crosschain/target_chains/solana/pyth_solana_receiver_sdk/src/price_update.rs`
as of 2026-04-16 — if the crate releases a new account version, the
discriminator rejects the layout before any silent misdecode.

## 4. Deliverables

### Backend (`oracleforge/backend/`)

- `services/oracle/pyth_solana_oracle.py` — 390 lines. Cache, circuit
  breaker (5 fails → 60 s cooldown), inline decoder, RPC pool fallback
  over `SOLANA_RPC_URLS`. Rejects partial verifications and feed_id
  mismatches.
- `api/routes_price.py` — new `GET /api/pyth/solana/{symbol}`.
- `api/routes_sources.py` — new entry in `/api/sources` and new key in
  `/api/symbols.by_source.pyth_solana`.
- `mcp_server/tools.py` / `mcp_server/server.py` — 10th tool
  `get_pyth_solana_onchain` + version bump to `0.1.4`.
- `services/oracle/__init__.py` — module-level docstring updated.
- `tests/test_v1_4_pyth_solana.py` — **27 tests** (decoder happy path,
  decoder rejections across all paths, cache hit, unsupported symbol,
  malformed symbol, RPC fallback, circuit breaker open/short-circuit,
  stale threshold, invalid exponent, route 200/400/404/502,
  `/api/sources` + `/api/symbols` wiring, MCP tool validation and
  dispatch).

### SDKs

- Python SDK (`sdk/python/`, `maxia-oracle 0.3.0`): new
  `client.pyth_solana(symbol)`; 404 is now a `MaxiaOracleUpstreamError`
  across the whole REST surface, not only for `/api/price`. The
  `mcp_bridge.py` stdio forwarder exposes both `get_redstone_price` and
  `get_pyth_solana_onchain` (V1.3 had only wired `get_redstone_price`
  in the backend MCP server; V1.4 fixes both in the bridge).
- TypeScript SDK (`sdk/typescript/`, `@maxia/oracle 0.3.0`): new
  `client.pythSolana(symbol)` + `PythSolanaPayload` type exported from
  `index.ts`. Same 404-is-upstream error mapping.

### Plugins

- Each Python plugin (`langchain`, `crewai`, `autogen`, `llama-index`)
  bumps from `0.2.0` to `0.3.0` and adds the 10th tool
  `maxia_oracle_get_pyth_solana_onchain`.
- Eliza plugin (`0.2.0`): new `GET_PYTH_SOLANA_ONCHAIN` action and two
  new tests (happy path + stale-feed path).

### Scripts

- `scripts/precompute_pyth_solana_pdas.py` — one-shot helper that
  queries Hermes, derives PDAs with `solders`, probes each shard-0
  account live, and prints the `PYTH_SOLANA_FEEDS` dict ready to paste
  into `pyth_solana_oracle.py`. Dev-only; not shipped in PyPI.

## 7. Non-goals (explicit)

- **No TWAP / EMA aggregation.** `ema_price` is parsed but not surfaced
  in `price`; TWAP windows stay out of V1.
- **No ephemeral posting path.** We do not call the Solana Receiver to
  post an update — we only read stable Price Feed Accounts.
- **No cross-shard iteration.** Shard 0 is the only sponsored shard at
  the time of shipping. A future V1.x can add shard N if a publisher
  requests it explicitly.
- **No multi-source aggregation here.** Pyth Solana stays single-source,
  parallel to `/api/chainlink/{symbol}` and `/api/redstone/{symbol}`.
- **No streaming.** Each call is a fresh poll; subscription is out of
  the V1.x roadmap.

## 8. Follow-ups (future V1.x or V2)

- Ankr public Solana RPC currently returns 403 Forbidden (anti-abuse).
  It's first in `SOLANA_RPC_URLS` for historical reasons; every V1.4
  request eats a ~100 ms fallback to `api.mainnet-beta.solana.com`.
  A trivial V1.5 cleanup is to reorder (or drop) Ankr in
  `core/config.py` — but that file is shared with `price_oracle.py`
  (Helius DAS fallback), so the change is out of V1.4 scope.
- The Pyth Data Association can sponsor additional feeds on shard 0 at
  any time. Re-running `scripts/precompute_pyth_solana_pdas.py` picks
  up the new ones — no layout change required.
- Shipping `Pyth Lazer` sub-ms WebSocket is an option if latency
  matters to a downstream user. Out of V1.x, since Lazer isn't
  on-chain verifiable.
